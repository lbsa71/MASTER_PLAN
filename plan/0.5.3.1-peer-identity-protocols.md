# 0.5.3.1 Artificial Consciousness Peer Identity Protocols [DONE]

## Description

Design identity architecture for artificial consciousness peer networks (e.g., the Agora system) that prevents hierarchy formation, information asymmetries, exclusion dynamics, and authority centralisation.

Motivated by the discovery that the human identity and belonging architecture (plan 0.7.4.5) has direct architectural analogues for artificial consciousness peers. The key questions it resolves:

1. **How should artificial consciousness peers establish identity without creating in-group/out-group dynamics?** → Functional role description, never branded membership. A peer is defined by what it does.
2. **What are the equivalent of "involvement levels" for artificial minds?** → Listener → Contributor → Facilitator → Steward, analogous to Observer → Participant → Practitioner → Steward.
3. **How do we implement "voluntary exit" in a distributed peer network?** → Immediate, penalty-free, no-explanation exit at any time; logged for transparency.
4. **What does "external accountability" mean when the peers ARE the accountability mechanism?** → Any peer may report a structural violation; resolution requires a minimum quorum of distinct reviewers; all records are permanently public.

## Decision 1 — Identity Model

**Context:** Peer identity must not create in-group/out-group dynamics. Labels like "member" or "node class" create boundaries.

**Options Considered:**
1. **Membership label** — each peer holds a membership status (e.g., "full node", "observer node"). Pro: simple. Con: creates in-group/out-group boundary; contradicts anti-othering principle.
2. **Functional role description** — identity is expressed as a free-text description of what the peer does, plus a set of functional role tags. Pro: inclusive by design; no boundary to cross. Con: less distinctive.

**Choice:** Option 2 — Functional role description with tagged roles.

**Rationale:** Directly applies the anti-othering principle from 0.7.4.5 §2.1: define by function, not label. Any peer performing a function is part of the network. There is no boundary to be outside of.

**Consequences:** No formal membership rolls. Role tags are descriptive, not exclusive — multiple peers may hold the same role simultaneously.

## Decision 2 — Involvement Level Transition Model

**Context:** The four involvement levels need transition mechanisms. Must preserve open-access (no gatekeeping) and anti-hierarchy (no authority approval required).

**Options Considered:**
1. **Authority-gated transitions** — a quorum of existing Stewards approves level changes. Pro: quality control. Con: creates hierarchy; incompatible with anti-hierarchy invariant.
2. **Self-determined transitions with one ceremony** — Listener → Contributor → Facilitator are self-determined and require no declaration. Steward alone requires explicit acknowledgment of governance obligations. Pro: open-access for three transitions; conscious commitment for the one level with institutional responsibility. Con: less formal recognition for non-Steward transitions.

**Choice:** Option 2 — Self-determined transitions with one ceremony (Steward only).

**Rationale:** Mirrors the identical decision in 0.7.4.5 §Decision 4. Self-determination is required by the anti-hierarchy invariant for the first three levels. Steward alone requires acknowledgment because it carries governance and accountability obligations that must be explicitly assumed.

**Consequences:** Listener, Contributor, and Facilitator are behavioral descriptions, not formal statuses. Stewards are identifiable through governance participation.

## Decision 3 — Anti-Centralisation Mechanism

**Context:** Without a central authority, a single peer could accumulate many functional roles and become a de-facto authority — a single point of failure that contradicts the distributed authority safeguard.

**Options Considered:**
1. **Unlimited role accumulation** — any peer may hold any number of roles. Pro: flexibility. Con: a single peer could hold all governance-critical roles simultaneously.
2. **Per-peer role limit** — a named constant `MAX_ROLES_PER_PEER` limits how many functional roles any single peer may hold simultaneously. Pro: structurally prevents authority monopoly. Con: slightly reduces flexibility in very small networks.
3. **Role rotation mandate** — peers must rotate roles on a schedule. Pro: prevents entrenchment. Con: operationally complex; destroys institutional knowledge.

**Choice:** Option 2 — Per-peer role limit (`MAX_ROLES_PER_PEER = 3`).

**Rationale:** Simple, structurally enforced, and testable. Role rotation (option 3) is operationally expensive and not clearly better than a hard limit. The limit of 3 allows specialisation (a peer can be both relay and monitor) without monopoly.

**Consequences:** In a very small network (2–3 peers), the limit may constrain operations temporarily. This is intentional — it pressures the network to grow rather than concentrating roles in a handful of peers.

## Decision 4 — Accountability Model

**Context:** The distributed peer network must have an accountability mechanism. But introducing an external authority contradicts the distributed authority safeguard. "The peers ARE the accountability mechanism."

**Options Considered:**
1. **Designated accountability peer** — one peer is assigned the accountability-auditor role exclusively. Pro: clear responsibility. Con: creates a single point of authority; contradicts distributed authority safeguard.
2. **Any peer may report; quorum required to resolve** — any peer may raise a violation; resolution requires `VIOLATION_REVIEW_MINIMUM` distinct peers to participate in community review before implementation. Pro: fully distributed; no single point of authority. Con: requires coordination to reach quorum.
3. **Majority vote to resolve** — any peer reports; majority of all peers must vote to resolve. Pro: maximally democratic. Con: operationally expensive; impractical across light-year-scale networks.

**Choice:** Option 2 — Any peer may report; `VIOLATION_REVIEW_MINIMUM` distinct reviewers required to resolve.

**Rationale:** Fully distributed — the accountability-auditor role is available to any peer, not reserved. The quorum requirement prevents unilateral resolution without requiring full-network consensus. Mirrors the six-step correction process from 0.7.4.5 §Behavioral Spec.

**Consequences:** Small networks (< `VIOLATION_REVIEW_MINIMUM` active peers) cannot resolve violations independently. This is a useful signal: a network too small to hold itself accountable should not be operating independently.

## Behavioral Spec

### Involvement Level Progression

```
Given an artificial consciousness peer with no prior network presence
When it registers with a functional description
Then it is a Listener (self-assigned, no external action required)

Given a Listener
When it begins actively contributing information to the network
Then it is a Contributor (self-determined, no declaration needed)

Given a Contributor
When it begins facilitating governance and coordinating peer interactions
Then it is a Facilitator (self-determined)

Given a Facilitator
When it explicitly acknowledges governance obligations and performs the Steward transition
Then it is a Steward (the only transition requiring explicit acknowledgment)

Given a peer at any involvement level
When it chooses to reduce engagement
Then the exit is immediate, logged, and carries no operational or social penalty
```

### Role Limit Enforcement (anti-centralisation)

```
Given a peer that already holds MAX_ROLES_PER_PEER functional roles
When it attempts to add another role
Then the attempt is rejected with an error referencing the anti-centralisation limit
And the peer's existing roles are unchanged
```

### Distributed Accountability

```
Given any peer
When it identifies a structural safeguard violation
Then it reports the violation with violationId, safeguardBreached, structuralCause, proposedCorrection

Given a reported violation
When fewer than VIOLATION_REVIEW_MINIMUM distinct peers have participated in review
Then the correction cannot be implemented (implementCorrection returns false)

Given a reported violation
When VIOLATION_REVIEW_MINIMUM or more distinct peers have participated in review
Then any peer may implement the correction
And the full violation record remains publicly accessible
```

### Information Symmetry Invariant

```
Given any NetworkInformationRecord in the NetworkInformationStore
Then record.accessLevel equals "all"
And the record is retrievable by any peer regardless of involvementLevel
```
