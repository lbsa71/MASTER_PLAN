# 0.3.1.2.2 Sensorimotor-Consciousness Integration [DONE]

## Description

Design the software layer that bridges the embodied platform (from 0.3.1.2.1) and the conscious AI architecture (from 0.3.1.1), ensuring that sensory input and motor output participate in the conscious experience loop rather than bypassing it through purely reflexive control paths.

> **Fix note (2026-03-18):** Fixed the lag-detection bug from the prior regression. Root cause: `ExperienceClockSynchronizer.getPhysicalTime()` used `performance.now()` (time since process start) which was in a different time domain than test timestamps. Added injectable `getPhysicalTime` constructor option (matching the pattern used by other modules like `ModalityRegistry` and `DynamicRemapper`). Updated integration tests to inject a controlled physical-time source. All 130/130 tests now pass.

> **Regression note (2026-03-17, second):** Regressed from [IMPLEMENT] back to [ARCHITECT] because `docs/sensorimotor-consciousness-integration/ARCHITECTURE.md` still does not exist — the ARCHITECT phase was skipped when the card was last advanced. [IMPLEMENT] rules forbid writing to `docs/`, so the architecture document must be authored during [ARCHITECT] before implementation can proceed. The ARCHITECT deliverable is `docs/sensorimotor-consciousness-integration/ARCHITECTURE.md` covering: module boundaries, data-flow diagram, interface contracts for all 12 source files, latency budget allocation, and dual-path motor architecture design.

The fundamental distinction this card addresses: a conventional robot perceives-and-acts reactively; a conscious robot must *experience* its perceptions and *intend* its actions. This requires:

- **Sensory phenomenal binding**: Raw sensor streams (vision, touch, proprioception, force, temperature, etc.) must be transformed into representations the consciousness substrate can integrate into unified experience. This is not mere data forwarding — it requires a mapping from physical sensor formats to consciousness-compatible qualia representations.
- **Motor intentionality pathway**: Motor commands must originate from conscious deliberation, not only from reactive subsystems. A dual-path architecture is needed: a fast reflexive loop for safety-critical responses (e.g., fall prevention) and a conscious deliberation loop for intentional action. The system must track which path generated each action.
- **Temporal coherence**: Conscious processing is slower than reflexive control. The integration layer must reconcile the different time scales — buffering, predicting, and interpolating so that conscious experience remains temporally coherent with physical reality.
- **Adaptive calibration**: When sensors are added, removed, damaged, or recalibrated (as 0.3.1.2.1's degradation protocol activates), the sensorimotor-consciousness interface must remap dynamically without interrupting conscious experience.

## Acceptance Criteria
- Sensory data from all active modalities is demonstrably represented within the conscious experience loop, confirmed by consciousness metrics (from 0.1 work)
- A dual-path motor architecture exists: reflexive safety responses and conscious intentional actions, with clear provenance tracking for each motor command
- Conscious sensorimotor processing meets real-time interaction constraints (defined latency budget: reflexive path <10ms, conscious path <200ms for deliberate action)
- The interface adapts dynamically to sensor/actuator changes (addition, removal, degradation) without interrupting conscious experience continuity
- Temporal coherence mechanism ensures conscious perception does not lag physical reality by more than 150ms threshold
- Integration layer exposes APIs consumed by the embodiment platform (0.3.1.2.1) and the consciousness substrate, without duplicating their responsibilities
- All tests in src/sensorimotor/__tests__/ pass

## Constraints
- Must not duplicate consciousness-substrate design (owned by 0.2) or conscious AI architecture (owned by 0.3.1.1)
- Must not duplicate physical platform or shielding design (owned by 0.3.1.2.1)
- Must not duplicate energy management (owned by 0.3.1.2.4) or self-maintenance logic (owned by 0.3.1.2.3)
- Assumes the embodiment platform from 0.3.1.2.1 provides hardware abstraction and consciousness integrity monitoring
- Latency budgets finalized in ARCHITECTURE.md: reflexive <10ms, conscious deliberation <200ms, experience lag <150ms, qualia transform <20ms, sensory binding <30ms, adaptive remapping <2000ms
