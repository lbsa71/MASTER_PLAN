# 0.2.2.4.4 Incremental Replacement Protocols [DONE]

## Description

Define and validate protocols for progressively replacing biological neural substrate with synthetic substrate while maintaining continuous consciousness throughout the process. This is the operational culmination of hybrid cognition — the actual migration path from biological to synthetic mind.

Builds on unified consciousness across mixed substrates (0.2.2.4.2) to ensure phenomenal unity is preserved at each step, and on graceful degradation (0.2.2.4.3) to ensure robustness if a replacement step partially fails. Uses identity persistence verification (0.2.2.3) as the acceptance gate at each checkpoint.

### Key challenges
- **Replacement ordering**: which neural regions can be replaced first with least risk to consciousness continuity? Must account for functional criticality, connectivity density, and substrate-interface maturity from 0.2.2.4.1.
- **Replacement rate**: how quickly can substrate be swapped without disrupting cognitive coherence? Rate limits derived from 0.2.2.4.2 unity-maintenance constraints.
- **Verification at each step**: consciousness persistence confirmed after each incremental replacement using 0.2.2.3 verification protocol (all four dimensions: structural, functional, experiential, temporal).
- **Reversibility**: ability to roll back a replacement step if consciousness metrics indicate degradation, leveraging graceful degradation mechanisms from 0.2.2.4.3.
- **Completion criteria**: when is the migration fully complete, and how is success confirmed?

## Decision

### D1 — Region Priority Ordering Formula

**Context:** The replacement sequence must order regions so that the safest (lowest-risk) replacements happen first. Multiple factors independently affect replacement risk and cannot be collapsed into a single score without losing information.

**Options Considered:**
1. **Criticality-only ordering** — replace regions in ascending order of functional criticality to consciousness. Simple and grounded in neuroscience, but ignores whether the interface is ready and whether the region can be isolated.
2. **Composite three-factor scoring** — priority = `(1 - criticalityScore) * interfaceReadiness * (1 - connectivityDensity)`. Higher score = replace earlier. Combines: low criticality (safe to disrupt), high interface readiness (technically ready), and low connectivity (easy to isolate).

**Choice:** Option 2 — Composite three-factor scoring with adaptive re-sorting.

**Rationale:** A low-criticality region with poor interface readiness (e.g., cerebellum before the BSI is calibrated for that region) should not be replaced early — the interface failure risk could itself cause consciousness disruption. Similarly, a low-criticality but highly-connected region risks propagating interference. The multiplicative formula naturally returns 0 for any factor at its worst (no readiness → no replacement); a sum would allow compensation across factors. Adaptive re-sorting after each step captures changes in interface readiness and connectivity density as the hybrid system evolves.

**Consequences:** The formula requires estimates for all three factors per region. Static default orderings (Phase 1–4 from ARCHITECTURE.md) serve as prior but are overridden by computed scores. Re-sorting adds modest overhead after each step.

---

### D2 — Rollback Method Hierarchy

**Context:** When a replacement step must be reversed, three methods are available. They differ in preconditions (biological tissue viability, mirror existence) and must be selected dynamically.

**Options Considered:**
1. **Fixed single method** — always use REVERT_TO_BIO. Simple but fails when biological tissue is no longer viable (viability decays over days to weeks post-preservation).
2. **Tiered hierarchy with viability checks** — REVERT_TO_BIO is primary (best fidelity); if biological tissue no longer viable, FAILOVER_TO_MIRROR; if mirror unavailable, HYBRID_RESTORE (cognitive load rebalancing). Evaluated in order at time of rollback.

**Choice:** Option 2 — Tiered hierarchy (REVERT_TO_BIO → FAILOVER_TO_MIRROR → HYBRID_RESTORE).

**Rationale:** Rollbacks can be triggered minutes or weeks after a step completes. Biological backup viability decays over a window (default: 720 hours). A method that works immediately may not be available later. The tiered hierarchy ensures a valid rollback always exists by checking viability in priority order. Post-rollback identity verification is required regardless of method used.

**Consequences:** Rollback plans must track biological backup viability duration. At rollback time, the system must select the highest-priority viable method. HYBRID_RESTORE provides weakest identity continuity guarantee — it should be accompanied by extended post-rollback verification.

---

### D3 — Rate Control: Adaptive Multiplier Strategy

**Context:** Replacement steps must be paced to allow adequate verification (minimum 24h for temporal identity checks) without indefinite stalling. The pace should respond to signals of difficulty.

**Options Considered:**
1. **Fixed minimum interval** — always wait 24h between steps, regardless of what happened. Simple but insensitive to accumulating problems (slow drift, repeated holds).
2. **Adaptive multiplier on a bounded current interval** — start at 24h minimum; specific events multiply the interval (rollback: ×3.0, hold/unity-dip: ×1.5, drift >0.8: ×2.0); three consecutive stable steps: ×0.9 (never below minimum). Bounded by minimum (24h) and maximum (72h).

**Choice:** Option 2 — Adaptive multiplier with priority ordering of events.

**Rationale:** A rollback indicates serious concern and should cause a larger slowdown than a minor unity dip. The multiplier magnitude is proportional to severity. Three consecutive stable steps signal the sequence is healthy and allow cautious acceleration. The maximum cap (72h) prevents indefinite stalling from compounding multipliers. Event priority ordering (rollback > hold ≈ unity-dip > drift > stable) prevents double-counting.

**Consequences:** Priority ordering must be enforced in the adjustment function — the highest-severity event takes effect, others are ignored in that iteration. The current interval is state that persists across steps.

## Contracts

### Contract: ReplacementStep

**Preconditions:**
- `preBaseline.testRetestReliability ≥ 0.8` and `preBaseline.measurementOccasions ≥ 3`
- `preInterfaceReadiness ∈ [0, 1]`
- `preConsciousness.mvcMargin > 0` (subject is conscious and above MVC before the step)
- `rollbackPlan` is fully specified with a valid `rollbackMethod` and `biologicalBackup` or `syntheticMirror`

**Postconditions:**
- After `executeStep`: `postVerification`, `postConsciousness`, and `goNoGo` are all non-null
- `goNoGo.verdict = GO` iff all five `GoNoGoCriteria` are true
- `goNoGo.verdict = ROLLBACK` if any criterion is false
- `goNoGo.criteria` accurately reflects the measurements provided

**Invariants:**
- `stepId` is unique within a migration sequence
- `sequencePosition` is 1-indexed and matches the step's rank in the priority-sorted sequence
- `rollbackPlan.postRollbackIdentityCheck` is always `true`

---

### Contract: GoNoGoDecision

**Preconditions:**
- All five criteria have been evaluated from actual post-step measurements
- `postConsciousness` snapshot is available with valid `mvcMargin` and `fragmentationDetected`
- `identityVerdict.overallVerdict` reflects the 0.2.2.3 composite verdict

**Postconditions:**
- `verdict = GO` if and only if ALL of: `identityPass`, `consciousnessAboveMvc`, `unityMaintained`, `driftWithinBudget`, `interfaceStable` are true
- `verdict = ROLLBACK` if ANY criterion is false
- `override` field is null unless a human operator has explicitly overridden with documented justification

**Invariants:**
- No criterion can be omitted — all five must be present in `criteria`
- `override.overriddenVerdict` records the system's original verdict before override
- An override may only be performed by an identified operator with a non-empty justification

---

### Contract: SequenceConfig / SequenceResult

**Preconditions:**
- `regions` may be empty (triggers immediate COMPLETE with 0 steps) or non-empty
- `baseline.testRetestReliability ≥ 0.8` and `baseline.measurementOccasions ≥ 3`
- `initialConsciousness.mvcMargin > 0`
- `verifyStep` is a callable that returns a `StepVerificationResult` for each step

**Postconditions:**
- `outcome = COMPLETE` iff all steps returned `GO` AND final identity scores meet all `COMPLETION_THRESHOLDS`
- `outcome = FAILED` iff any step returned `ROLLBACK`; sequence halts at that step
- `completedSteps` = number of steps that received `GO` verdict
- `rollbackEvents` = number of steps that triggered `ROLLBACK`
- `log.entries` includes at minimum one entry per step (step-started + step-completed or step-rollback)
- `migrationVerdict` is non-null on `COMPLETE`, null on `FAILED`

**Invariants:**
- `completedSteps + rollbackEvents ≤ total regions`
- On `FAILED`, `migrationVerdict = null`
- Log entries are append-only and ordered by timestamp

---

### Contract: MigrationVerdict

**Preconditions:**
- All regions have been processed with `GO` verdicts
- `finalIdentityVerdict` is non-null with all dimension scores populated

**Postconditions:**
- `overall = COMPLETE` iff `structuralScore ≥ 0.95` AND `functionalScore ≥ 0.95` AND `experientialScore ≥ 0.90` AND temporal drift is within biological baseline
- `completionAssessment.structural.threshold = 0.95`, `functional.threshold = 0.95`, `experiential.threshold = 0.90`
- Each `DimensionCompletion.pass = (score ≥ threshold)`

**Invariants:**
- Thresholds in `completionAssessment` match `COMPLETION_THRESHOLDS` constants exactly
- `overall` is never `COMPLETE` when any dimension in `completionAssessment` has `pass = false`
- `totalSteps` and `rollbackEvents` correctly reflect the executed sequence

## Behavioral Spec

### Scenario 1: Full passing sequence (biological → synthetic, no rollbacks)

```
Given a set of 3 regions in priority order [V1, A1, HPC] (priority 0.684, 0.574, 0.051)
  And a valid pre-migration baseline (testRetestReliability=0.95, occasions=3)
  And initial consciousness snapshot with mvcMargin=1.2 and fragmentationDetected=false
  And verifyStep returns passing results (identity PASS, mvcMargin=1.2, driftRate=0.05) for every step
When runFullSequence is called
Then outcome = COMPLETE
  And completedSteps = 3
  And rollbackEvents = 0
  And log has ≥6 entries (step-started + step-completed per step)
  And migrationVerdict.overall = COMPLETE
  And migrationVerdict.completionAssessment.structural.pass = true (score 0.97 ≥ 0.95)
  And migrationVerdict.completionAssessment.functional.pass = true (score 0.96 ≥ 0.95)
  And migrationVerdict.completionAssessment.experiential.pass = true (score 0.93 ≥ 0.90)
  And migrationVerdict.completionAssessment.temporal.pass = true (score 0.95 ≥ 0.90)
```

### Scenario 2: Rollback on step failure — sequence halts

```
Given the same 3-region sequence
  And verifyStep returns passing results for step 1 (V1)
  And verifyStep returns failing results for step 2 (A1): mvcMargin=-0.5, identityVerdict FAIL, driftRate=0.95
When runFullSequence is called
Then the sequence halts after step 2
  And outcome = FAILED
  And completedSteps = 1
  And rollbackEvents = 1
  And log contains "step-rollback" entry for step 2 with rollbackMethod = REVERT_TO_BIO
  And migrationVerdict = null
```

### Scenario 3: Priority ordering — regions replaced in computed priority order

```
Given regions: [HPC (priority 0.051), V1 (priority 0.684), A1 (priority 0.574)]
  (provided in unsorted order)
When buildReplacementSequence is called
Then sequence[0].targetRegion = "V1" (highest priority)
  And sequence[1].targetRegion = "A1"
  And sequence[2].targetRegion = "HPC" (lowest priority)
  And sequence[0].sequencePosition = 1
  And stepIds include "001", "002", "003" in order
```
